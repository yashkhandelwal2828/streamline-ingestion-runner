    Dashboard → streamline-api → Settings → Start Command must be npm start, not npm run start:integrated or start:all.
  2. Set RUN_BACKGROUND_WORKER_IN_API=false
    Already in render.yaml. Verify it is set in the live Render Dashboard env vars (not overridden).
  3. Shrink API database connection pool
    Append to Render DATABASE_URL:
    postgresql://USER:PASS@HOST/DB?sslmode=require&connection_limit=4
    Neon free tier: keep API at 4 connections so GH runner + API don't exhaust the pool.
  4. Merge backend worktree changes to main (when ready)
    Branch codex-ingestion-pipeline has: queueConfig.ts, enqueue scripts, render.yaml, worker cron guard. Deploy after review.
  5. Do not use drain from GitHub
    Admin POST /api/ingest/drain may stay for manual ops only. CI must never call it.

Phase 3 Enable production scheduling

Turn on cron workflows in order after dry run succeeds.

  1. Enable process-queue cron (every 45 min)
    Already in process-queue.yml. Merging to main activates it. Monitor first 2–3 runs.
  2. Enable enqueue-daily (10 */6 * * *)
    4×/day catalog top-up. Skips if daily target met, active catalog job exists, or queue above 10k watermark.
  3. Enable enqueue-delta (20 */6 * * *)
    Recent releases lane; runs 10 min after daily enqueue each cycle.
  4. Enable maintenance
    02:30 UTC — snapshot + job history cleanup
    03:00 UTC — OMDb DEEP_ENRICH enqueue (up to ~950/day)
  5. Switch Render to API-only (if not already)
    Redeploy API after confirming GH processor is claiming jobs. This removes ingestion CPU/memory from Render.
First 24h: Watch queue depth, TMDB 429s in logs, OMDb daily usage, net-new Movie rows per day.

Phase 4 Backfill tuning (after stable week)

  1. Raise daily target gradually
    Start INGESTION_DAILY_TARGET=20000. If queue stays under watermark and DB handles writes, try 30k–40k via workflow env or repo variables.
  2. Tune TMDB rate if needed
    TMDB_REQUESTS_PER_SECOND default 8. If frequent 429s, lower to 5. If clean for days, try 10.
  3. Tune worker burst duration
    timeout 300 in process-queue.yml. Increase to 420–600 only if queue backlog grows and rate limits stay clean.
  4. Keep OMDb independent
    Do not raise INGESTION_OMDB_DAILY_LIMIT above 1000. Missing ratings during catalog catch-up is expected.
  5. Schema sync discipline
    When backend prisma/schema.prisma changes, copy to runner repo and redeploy both.
